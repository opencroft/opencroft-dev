# Extension system

Everything you can drag onto the canvas is a node from an **extension**.
Extensions are loadable bundles with two halves:

- A **client bundle**: React components for the canvas tile, inspector
  tabs, command-bar modes, settings pages.
- A **server module**: `actions`, `nodeActions` and `exposeOutput`
  callbacks that run inside Next.js with full Node.js access.

Both halves live in the same folder, share an `extension.json` manifest, and
are compiled by esbuild into a `dist/` subfolder. The runtime loads the
client bundle into the browser and `eval`s the server bundle inside Next.js.

## Where extensions live

| Scope | Folder | Notes |
| --- | --- | --- |
| `builtin/<slug>` | `app/(extension-runtime)/_builtin/<slug>/` | Shipped with the app. Currently only `core`. |
| `local/<slug>` | `data/extensions/local/<slug>/` (or `OPENCROFT_EXT_ROOT`) | User-authored. Created by the editor or by MCP. |

Extension IDs are always `<scope>/<slug>` and must match the folder name; the
manifest validator rejects any mismatch
([_server/manifest.ts:17](../%28extension-runtime%29/_server/manifest.ts#L17)).

Path helpers in [_server/paths.ts](../%28extension-runtime%29/_server/paths.ts):
`extDir(id)`, `extDistDir(id)`, `extDistFile(id, name)`,
`localExtRoot()`, `builtinExtRoot()`.

## Folder layout

A typical extension:

```
data/extensions/local/my-thing/
├── extension.json        # the manifest
├── src/
│   └── client.tsx        # default export → defineExtension({ ... })
├── server/
│   └── index.ts          # exports actions, nodeActions, exposeOutput
└── dist/                 # generated by the compiler
    ├── client.js
    └── server.js
```

`dist/` is excluded from mtime walks (so it doesn't trigger its own rebuild)
and from local-extension file listings used by the editor.

## The manifest

Schema in [_types.ts](../%28extension-runtime%29/_types.ts):

```ts
interface ExtensionManifest {
  id: string;                            // "<scope>/<slug>", must match folder
  name: string;
  version: string;                       // semver, validated by validateManifest
  description?: string;
  extensionDependencies?: string[];      // activated transitively before this one
  nodes?: NodeMetadata[];                // static metadata for palette discovery
  contexts?: ExtensionContextType[];     // typed handles this extension introduces
  main?: string;                         // entry override (rare)
  exports?: { server?: string; client?: string }; // entry overrides (rare)
  activationEvents?: string[];           // reserved
}

interface NodeMetadata {
  typeId: string;
  name: string;
  category?: string;
  description?: string;
  icon?: string;          // lucide-react icon name
  accent?: string;        // OKLCH color string
  handles?: ExtensionHandle[];
  actions?: NodeAction[];
  defaultData?: Record<string, unknown>;
}

interface ExtensionHandle {
  id: string;
  contextType: string;     // e.g. "terminal-context"
  role: 'source' | 'target';
  label?: string;
}

interface NodeAction {
  id: string;
  label: string;
  description?: string;
  icon?: string;
  inputSchema?: Record<string, unknown>;
}
```

The manifest is parsed by `readManifest(extensionId)`
([_server/manifest.ts:9](../%28extension-runtime%29/_server/manifest.ts#L9)).
Validation only enforces `id`, `version` and the id-matches-folder rule.
Everything else is optional.

`nodes[]` in the manifest is the **palette** view: enough metadata to render
the left sidebar without compiling anything. The runtime view (React
components, `exposeOutput`, inspector tabs) is added later when the client
bundle calls `defineExtension({ nodes: [...] })` at load time. The two lists
are joined by `typeId`.

## The client bundle

Authored as `src/client.tsx`. Imports `@ext/host` (the host API) and
`@ext/ui` (shadcn primitives). Default-exports the result of
`defineExtension(...)`:

```tsx
import { defineExtension, NodeFrame, OutputHandle, icons } from '@ext/host';

const MyNode = ({ data, selected }) => (
  <NodeFrame icon={icons.Box} title={data.title} selected={!!selected}>
    <OutputHandle type='terminal-context' id='out'>
      <span className='text-xs'>terminal</span>
    </OutputHandle>
  </NodeFrame>
);

export default defineExtension({
  manifest: { id: 'local/my-thing' },
  nodes: [{
    typeId: 'my-thing-node',
    name: 'My Thing',
    icon: 'Box',
    component: MyNode,
    defaultData: { title: 'Hello' },
    exposeOutput: (handleId, data) => ({ kind: 'fake', payload: data }),
  }],
  contexts: [],          // optional. Only if you introduce new context types
  commandModes: [],      // optional. Registers ⌘K-style modes
  settings: [],          // optional. Pages shown in /settings
});
```

What `defineExtension` returns
([_client/host.ts:148](../%28extension-runtime%29/_client/host.ts#L148)) is
the `ExtensionDeclaration` the registry will store. It includes
`NodeDefinition`s with `component`, `inspector`, `inspectorTabs[]` and an
optional `exposeOutput` (used by the registry to flow data through the
client at render time. The **server-side** `exposeOutput` lives in the
server module).

### `@ext/host` API

`extensionHostApi`
([_client/host.ts:310](../%28extension-runtime%29/_client/host.ts#L310))
exposes:

- React itself (`React`, hooks, `createElement`, `Fragment`, etc.).
- `defineExtension`, `NodeFrame`, `NodeCard`, `NodeCardHeader`,
  `NodeCardContent`, `NodeResizer`, `Handle`, `Position`, `InputHandle`,
  `OutputHandle`.
- `useNodeContext`, `inspectorIntent`, `useInspectorIntent`,
  `useGraphNodes`, `useGraphEdges`, `useReactFlow`,
  `useUpdateNodeInternals`.
- `callAction(extensionId, action, args)`: invoke a server action exported
  by any extension.
- `callNodeAction(nodeId, actionId, params?)`: dispatch a manifest action.
- `createStorage(extensionId, namespace?)`: namespaced key/value store
  backed by the `extension-storage` setting row.
- `createPortal`, `icons` (all of `lucide-react`), `toast` (sonner).
- Stream helpers: `getStream`, `subscribe`, `broadcast`, plus the `Stream`,
  `TextChunk`, `AudioChunk` types.

### `@ext/ui` API

`extensionUiApi`
([_client/host.ts:276](../%28extension-runtime%29/_client/host.ts#L276)) is
a curated subset of shadcn primitives: Badge, Button, Input, ControlledInput,
Label, Flex, ScrollArea, Select (+ its parts), Separator, Textarea, Slider,
ChatMessage, ChatInput, StatusIndicator, Dialog (+ its parts), FileBrowser,
FileManagerProvider, CommandBar (+ its parts).

These are surfaced explicitly so an extension never has to depend on a
specific shadcn build of the host. New primitives are added by extending
this object.

## The server module

Authored as `server/index.ts`. Default-exports, or named-exports, any of:

```ts
export const actions: Record<string, (...args: unknown[]) => Promise<unknown>>;
export function exposeOutput(handleId, nodeData, typeId, nodeId): unknown;
export const nodeActions: Record<typeId, Record<actionId, (ctx: NodeActionCtx) => Promise<unknown>>>;
```

The host gives every server module access to a per-extension `host` object
(see [_server/host.ts:230](../%28extension-runtime%29/_server/host.ts#L230)):

```ts
host.fs       // node:fs/promises
host.os       // node:os
host.path     // node:path
host.exec(cmd)            // shell.ts wrapper, returns stdout
host.execFile(cmd, args)
host.cacheDir(...parts)   // resolves to <project>/.cache/extensions/<id>/<parts>
host.crypto.encrypt / decrypt
host.prisma               // shared Prisma client
host.settings.get / set
host.graph.listNodes / getNode / listNodesByType / listEdges /
           updateNode / createNode / deleteNode
host.storage.get / set / delete / list / clear   // namespaced key/value
```

The host is exposed to the bundle via `globalThis.__extensionServerApi`,
which the compiler shim reads at load time. Extension code does not import
the host directly; it uses the `@ext/host` import (rewritten by the compiler
to `globalThis.__extensionServerApi.host`).

`actions[name]` is invoked by the **client** through `callAction`
(`/api/ext/[scope]/[slug]/action/[name]/route.ts`).

`nodeActions[typeId][actionId]` is invoked by `dispatchNodeAction(nodeId,
actionId, params)` ([_server/node-actions.ts:208](../%28extension-runtime%29/_server/node-actions.ts#L208)).
The `NodeActionCtx` argument carries:

- `nodeId`, `typeId`, `data`, `params`.
- `input(handleId)` / `inputSource(handleId)`: read resolved contexts.
- `connectedSources(handleId)`: every node connected to this handle, raw.
- `containingNodes(typeId?)`: the surrounding frame nodes (e.g. the
  `Section` or `Domain` a node sits inside).

If the handler throws, the error message is written to
`node.data.__errors` and the throw is re-raised so the inspector can show a
red banner. A successful call clears `__errors` again.

## The compiler

[_server/compiler.ts](../%28extension-runtime%29/_server/compiler.ts) wraps
esbuild. Two builds per extension:

| Bundle | Format | Externals | Output |
| --- | --- | --- | --- |
| `client` | ESM, browser platform | `@ext/host`, `@ext/ui`, `react`, `react-dom`, `react/jsx-runtime` (rewritten to a host shim) | `<ext>/dist/client.js` |
| `server` | CJS, node platform | `node:*`, `fs`, `path`, `os`, `child_process`, `crypto`, `stream`, `util`, `events`, `ssh2`, `@prisma/client`, `@prisma/adapter-better-sqlite3` | `<ext>/dist/server.js` |

The `hostVirtualPlugin` rewrites every import of `react`, `react-dom`,
`@ext/host` and `@ext/ui` into a tiny shim that pulls those modules from
`globalThis.__extHost` (client) or `globalThis.__extensionServerApi`
(server). The point is twofold:

- React is never duplicated, so hooks, context and Suspense work the same in
  every extension.
- The host can swap its `defineExtension`/`NodeFrame`/etc. without
  recompiling extensions.

`buildExtension(id, manifest)` returns:

```ts
interface BuildResult {
  success: boolean;
  errors: CompileError[];
  warnings: CompileError[];
  clientHash: string;
  serverHash: string;
}
```

Client and server hashes are content hashes used to bust the
`<script src="/api/ext/.../client.js?v=...">` cache.

## The loader

[_server/loader.ts](../%28extension-runtime%29/_server/loader.ts) is
responsible for:

1. **Source mtime walk.** `sourceMtime(id)` reads the recursive max mtime of
   `extension.json`, `package.json`, `src/`, `server/` (skipping
   `node_modules` and `dist`).
2. **Rebuild on stale.** `ensureBuilt(id, manifest)` compares source mtime to
   the bundle mtimes; if newer, it calls `buildExtension`. If the build
   fails but a previous bundle exists, it logs and keeps the old one. If the
   build fails and there's nothing to fall back to, it throws.
3. **Activation.** `activate(id)` reads the manifest, recursively activates
   `extensionDependencies`, ensures the bundle is built, then `eval`s the
   server bundle inside a `Function('module', 'exports', 'require', ...)`
   wrapper so the bundle gets a real `require`. The result is cached in
   `globalThis.__EXT_MODULE_CACHE__`.
4. **Cache invalidation.** `flushCache(id?)` drops one entry or the whole
   cache; the editor and the MCP `compile_extension` tool use it after
   writes.

The first call from the canvas, the MCP server or any node action triggers
`getExtensionModule(id)`, which in turn triggers `activate(id)` if the
cache is cold or stale. Build errors surface as toasts via
`toastStore.broadcast({ type: 'toast', toastType: 'error', ... })`.

## The registry

[_client/registry.ts:54](../%28extension-runtime%29/_client/registry.ts#L54)
is the client-side index. It stores:

- `byExtensionId: Map<string, ExtensionDeclaration>`
- `byTypeId: Map<string, { extension, nodeIndex }>`: for `resolveNode`.
- `contextTypes: Map<string, ExtensionContextType>`: color and label per
  context type (used to color handle pins).
- `commandModes: Map<string, CommandModeDefinition>`: every command-bar
  mode contributed across extensions.

`resolveNode(typeId)` returns a `ResolvedNode` with the icon, accent,
defaults, React components and `exposeOutput`. The canvas uses this to build
React Flow's `nodeTypes` map.

## The editor

UI: `app/(extension-editor)/_components/extension-editor.tsx`. It opens
in-place inside the canvas (toggled by the `ViewMode` state in
`flow-editor.tsx`). Server actions live in
[_actions/local-extensions-actions.ts](../%28extension-editor%29/_actions/local-extensions-actions.ts):

| Action | Behaviour |
| --- | --- |
| `listLocalExtensions()` | Walks `data/extensions/local/`, returns full file maps. |
| `getLocalExtension(id)` | Reads one extension. |
| `createLocalExtension(files)` | Refuses if the folder exists; then writes every file. |
| `updateLocalExtension(id, files)` | Writes every file in the map (no diff. Caller sends the full set). Flushes the loader cache. |
| `deleteLocalExtension(id)` | `rm -rf` on the folder, flushes the cache. |
| `compileLocalExtension(id)` | Forces a build and returns `BuildResult`. |

A starter template is generated by
[_templates/template.ts](../%28extension-editor%29/_templates/template.ts):
an `extension.json` with one node, a `src/client.tsx` that draws a blank
`NodeFrame`, and a `server/index.ts` that exports a single `ping → 'pong'`
action.

`watch-extensions.ts` (`scripts/watch-extensions.ts`) is the dev-time
filewatcher that calls `compileLocalExtension(id)` whenever any file under
`data/extensions/local/<slug>/` changes outside of `dist/`.

## Hot reload

Two paths:

- **From the editor.** `updateLocalExtension` flushes the loader cache; the
  next dispatch will re-activate. The editor also broadcasts a
  `extensions_updated` SSE event so every open canvas reloads its bundles.
- **From watch-extensions.** Same thing: flush the cache and broadcast.

The canvas listens for `extensions_updated` in `useGraphEvents` and calls
`loadAllExtensions()` again. Re-registration replaces every entry in the
registry; the next render of the canvas uses the new components. Existing
node state is preserved because graph state is stored at the React Flow
level, not the registry level.
